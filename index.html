01 Computer & System Fundamentals

CPU, memory, throughput, latency, scaling — how these concepts apply to Azure integration design.

Core Concepts

Throughput vs Latency

Throughput = how many operations per second the system can handle. Latency = how long a single operation takes. These trade off: batching increases throughput but raises latency. In Azure Service Bus, throughput is measured in messages/sec; latency is the round-trip time for a single send+receive.

For integration design: a document exchange system might prioritise throughput (bulk file processing via Event Hub), while a payment approval flow prioritises low latency (synchronous APIM → Function App → DB).

Horizontal vs Vertical Scaling

Vertical scaling = bigger machine (more CPU/RAM). Limited, requires downtime. Horizontal scaling = more instances. Azure's default model — Functions scale out automatically, Service Bus Premium scales via messaging units, Event Hub scales via throughput units.

In integration: Logic Apps Standard uses horizontal scaling on Azure App Service Plans. Functions on Consumption plan scale to zero and out to hundreds of instances automatically based on queue depth.

Stateless vs Stateful

Stateless: each request is independent, no server-side session. Easy to scale horizontally. HTTP endpoints, most APIs. Stateful: maintains state across calls. Logic Apps Standard stateful workflows persist to Azure Storage. Durable Functions persist orchestration state to Azure Table Storage. Service Bus sessions provide ordered, stateful message processing.

CAP Theorem in Azure Context

A distributed system can guarantee at most 2 of 3: Consistency (all nodes see same data), Availability (every request gets a response), Partition tolerance (system works despite network splits). Azure Storage is typically AP (highly available, eventually consistent). Service Bus is CP — it prioritises consistency of message ordering over availability during partitions. APIM with geo-redundancy trades some consistency for availability.

Interview Q&A

How would you design an Azure integration solution for high throughput message processing? ▼

For high throughput I'd reach for Azure Event Hub as the ingestion layer — it's designed for millions of events per second with partitioned parallel processing. I'd pair it with Azure Stream Analytics or Azure Functions (Event Hub trigger) as the processor, scaling out across all partitions via consumer groups. If the downstream needs reliable delivery semantics (at-least-once, dead-lettering), I'd fan out from Event Hub into Service Bus topics for further routing. On the Service Bus side I'd use Premium tier for predictable performance, size the messaging units based on expected TPS, and use batched send operations from the producers to maximize throughput. I'd monitor using Service Bus metrics in Azure Monitor — specifically the incoming message rate and server-side latency — to tune accordingly.

Explain the difference between latency and throughput with a real Azure example. ▼

Latency is how long one operation takes; throughput is how many operations you can do per second — they often trade off. A practical Azure example: when I configured APIM to publish messages directly to Service Bus using the managed identity send-request policy, I measured end-to-end latency of a single publish at around 50-80ms including APIM policy execution. But throughput — the system could handle several thousand such requests per minute on Standard tier. If I needed to push throughput higher I'd batch messages in APIM policies and send to Service Bus in bulk, which increases per-message latency slightly but multiplies overall throughput. For time-sensitive flows like fraud checks I optimise for latency; for bulk data ingestion I optimise for throughput and accept higher per-message delay.

How does Azure Functions handle scaling and what are its limits? ▼

Azure Functions on the Consumption plan scales dynamically: the host monitors the trigger source (queue depth, event rate) and adds instances as load increases, up to 200 instances by default. Each instance is stateless, processes messages independently. The scale controller checks queue length every few seconds and scales aggressively when needed. The limitation is cold start — on Consumption, an idle instance is deallocated, so the first request after idle periods incurs a cold start (200-500ms for .NET, longer for some runtimes). For integration scenarios where cold start is unacceptable I use Premium plan which keeps warm instances ready. For Service Bus triggered Functions, the scale is driven by the number of messages in the queue — the runtime will scale up to one instance per message up to the concurrency limit. I've used this extensively on the Insurance project where Function Apps processed claims documents triggered by Service Bus queue messages.

What is idempotency and why is it critical in messaging systems? ▼

Idempotency means applying the same operation multiple times produces the same result as applying it once. In messaging systems this is critical because at-least-once delivery is the norm — the broker guarantees delivery but may deliver duplicates (during network failures, retries, or consumer restarts). If your message processor is not idempotent, a duplicate message causes a duplicate database insert, duplicate API call, or duplicate payment — all catastrophic. To implement idempotency: include a unique message ID in every message (Service Bus exposes MessageId), on the consumer side check if that ID has already been processed (typically via a database deduplication table or cache), and skip reprocessing if already seen. Service Bus also has built-in duplicate detection at the namespace level: enable it on a queue/topic with a detection window (1-10 minutes), and it will reject messages with duplicate MessageIds. On the Insurance project I implemented idempotent claim processors using message correlation IDs stored in SQL — this protected us against the at-least-once redelivery behaviour during the migration.

02 Networking & Web Protocols

HTTP, REST, SOAP, Webhooks — how APIs communicate and why it matters for integration.

Core Concepts

HTTP/HTTPS & TLS

HTTP is stateless request-response protocol. HTTPS adds TLS encryption: client sends ClientHello with supported cipher suites, server responds with certificate, keys are exchanged (ECDH), symmetric session key is derived, all traffic encrypted. In Azure, TLS 1.2+ is enforced across APIM, Function Apps, Logic Apps HTTP triggers. APIM can terminate TLS and re-encrypt to backend (SSL offloading) or pass through.

REST vs SOAP

REST: architectural style using HTTP verbs (GET/POST/PUT/DELETE/PATCH), resources identified by URIs, JSON payloads, stateless. SOAP: protocol with XML envelope/body/header, WSDL contract, transport-agnostic (but usually HTTP). SOAP has built-in standards for security (WS-Security), reliability (WS-ReliableMessaging), transactions. In BizTalk we consumed SOAP/WCF services extensively. In the Azure migration, I exposed REST APIs via APIM, translating SOAP backends to REST for modern consumers — APIM has built-in SOAP pass-through and SOAP-to-REST transformation via policies.

Webhooks

A webhook is an HTTP callback — the provider calls your endpoint when an event occurs (push model vs REST polling pull model). Logic Apps HTTP trigger is essentially a webhook endpoint. Event Grid delivers events to webhook endpoints. Benefits: real-time, no polling overhead. Challenge: your endpoint must be publicly reachable, must respond within timeout (30s for Event Grid), must handle retries. I used webhook-triggered Logic Apps in the Insurance project to receive policy status updates from a third-party system.

Interview Q&A

How does APIM handle the difference between REST and SOAP backends?▼

APIM can import WSDL directly and expose the SOAP service as-is (SOAP pass-through) or transform it to a REST interface. In the REST-to-SOAP mode, APIM accepts JSON from clients, and the inbound policy transforms it to a SOAP XML envelope using set-body with liquid templates before forwarding to the backend. The response is then transformed back. On our Insurance project we had legacy SOAP policy management services — I configured APIM REST facades over them so our Logic Apps could call JSON REST endpoints while APIM handled the SOAP translation transparently. This meant no changes to the legacy backend and modern REST interfaces for all new integration consumers.

What is the difference between synchronous and asynchronous APIs? When would you use 202 Accepted?▼

Synchronous: client waits for the response before continuing — request-response, blocking. Good for quick lookups, validations. Asynchronous: client fires request and gets back immediately, processes result later via polling or callback. The 202 Accepted status code signals an async pattern: the server accepted the request for processing but hasn't completed it yet. Typically paired with a Location header pointing to a status endpoint the client can poll. In Azure, APIM has built-in support for the async operation pattern. I've used this pattern for claims ingestion on the Insurance project — the API call accepted the claim payload and returned 202 with a correlation ID, then the actual processing happened asynchronously via Service Bus and a Function App, and the client could poll a status endpoint. This decoupled the client from the processing latency and made the system far more resilient.

03 Identity & Security

OAuth 2.0, JWT, Azure Entra ID, Managed Identity, RBAC, Key Vault — the security backbone of Azure integration.

OAuth 2.0 Flows

Client Credentials Flow (most common in integration)

App → POST /token
  grant_type=client_credentials
  client_id=xxx
  client_secret=xxx
  scope=https://servicebus.azure.net/.default

← { access_token: "eyJ...", expires_in: 3599 }

Used for service-to-service (no user). App authenticates with its own identity. This is what Managed Identity replaces — instead of client_id + secret, Azure handles token acquisition internally.

  → Call Service Bus with Bearer token

System-assigned: tied to the resource lifecycle, auto-deleted when resource is deleted. User-assigned: standalone identity, can be shared across resources, survives resource deletion. Interview Q&A

How did you implement secure credential management across your integration solution?▼

I implemented zero-trust credential management across all EY Insurance integration components. The principle was: no secrets in code, config files, or environment variables. For service-to-service calls within Azure — Logic Apps to Service Bus, Functions to Storage, APIM to Service Bus — I used Managed Identities with RBAC role assignments. For example, the Function App's system-assigned identity was granted the "Azure Service Bus Data Sender" role on the Service Bus namespace. For third-party credentials (SFTP passwords, external API keys) that had to exist somewhere, I stored them in Azure Key Vault and referenced them via Key Vault references in App Settings — Azure resolves the reference at runtime using the app's managed identity. This meant even application settings were secret-free in the portal. APIM used Named Values backed by Key Vault references for API subscription keys and backend credentials. The result was a fully auditable, rotation-friendly credential setup with no hardcoded secrets anywhere in the stack.

Explain the difference between Authentication and Authorization. How does RBAC enforce authorization in Azure?▼

Authentication (AuthN) answers "who are you?" — verifying identity via credentials, tokens, certificates. Authorization (AuthZ) answers "what are you allowed to do?" — checking permissions after identity is established. Azure RBAC is the authorization system. It has three components: Security Principal (user, group, service principal, or managed identity), Role Definition (collection of permissions like read/write/delete on specific resource types), and Scope (management group, subscription, resource group, or resource). When an action is requested, Azure RBAC checks: does this principal have a role assignment at this scope (or parent scope) that includes this action? For integration: I assign "Azure Service Bus Data Sender" to a Function App's managed identity at the namespace scope — this allows it to send to any queue/topic in that namespace. If I needed to restrict it to a specific queue, I'd scope the role to that queue resource. RBAC denies by default — explicit assignments are required.

When would you use a user-assigned managed identity vs system-assigned?▼

System-assigned is simpler — it's created with the resource and auto-deleted when the resource is deleted. Good for single-purpose, isolated resources. User-assigned is a standalone identity that you create independently and can assign to multiple resources. I use user-assigned when: (1) multiple resources need the same identity and permissions — e.g., a fleet of Function Apps that all send to the same Service Bus, I create one user-assigned MI, assign it to all Functions, and manage RBAC in one place; (2) during blue-green deployments where a new resource needs the same identity as the old one before cutover; (3) when the identity needs to outlive any specific resource. On the Insurance project I used system-assigned for simplicity since each integration component had distinct responsibilities. If I were redesigning it today I'd use user-assigned for the set of components that share Service Bus access, for easier RBAC management.

HA vs DR

High Availability (HA)

Keeps the system running during partial failures within a region. Azure Availability Zones are physically separate datacenters in a region (power, cooling, networking independent). Service Bus Premium, APIM Premium support zone-redundant deployment — spread across 3 AZs automatically. Logic Apps Standard on zone-redundant App Service Plans. Target: 99.99% SLA.

Disaster Recovery (DR)

Recovery from full regional outage. Active-Passive: primary region runs, secondary is warm standby. Failover triggered manually or via Traffic Manager. Service Bus Geo-Disaster Recovery replicates namespace metadata (not messages) to secondary — switch alias in minutes. APIM Premium multi-region deploys gateway in multiple regions with Azure Traffic Manager routing. RTO and RPO are the key metrics: RTO (Recovery Time Objective) = how quickly can you recover; RPO (Recovery Point Objective) = how much data can you lose.

ARM & Resource Manager

Azure Resource Manager is the management layer for all Azure resources. All operations (portal, CLI, SDK, Terraform) go through ARM. ARM enforces RBAC, provides deployment templates (ARM JSON / Bicep), and manages resource lifecycle. For integration deployments I use Bicep — it compiles to ARM JSON but is much more readable. A typical integration deployment Bicep would provision: Resource Group → Service Bus namespace + queues/topics → Storage Account → Function App → App Service Plan → APIM → Key Vault → all RBAC role assignments.

Tags for governance

Tags are key-value metadata on resources. Critical for: cost allocation (tag by project/team/env), automation (identify resources for automated ops), compliance (tag data classification). In EY deployments I tagged all resources with environment, project, cost-centre, and data-classification tags enforced via Azure Policy.

Interview Q&A

How would you design an HA/DR strategy for an Azure Integration solution?▼

I approach this in two layers: HA for in-region resilience, DR for cross-region recovery. For HA: deploy Service Bus Premium with zone redundancy across all 3 AZs in the region, deploy APIM Premium with zone redundancy, deploy Function Apps on Premium plan with zone-redundant App Service Environment, and use zone-redundant Storage for Logic Apps Standard state. This gives 99.99% SLA for normal single-region operations. For DR: I configure Service Bus Geo-Disaster Recovery pairing to a secondary region — the alias is the connection string, so failover just requires triggering the alias switch, no application config changes. APIM Premium gets a secondary region gateway added via multi-region deployment; Traffic Manager or Front Door routes to the healthy region. For Logic Apps, I maintain the same workflow definition deployed in both regions (IaC handles this), with the secondary in standby mode. The RPO for Service Bus Geo-DR is near-zero for namespace config but potentially minutes for in-flight messages — for zero-RPO message requirements I'd use Service Bus Premium with Active-Active in both regions and application-level duplicate detection to handle both regions receiving the same messages.
